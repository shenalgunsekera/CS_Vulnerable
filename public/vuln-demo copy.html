Security Vulnerability Demonstrations

This page demonstrates two real-world vulnerabilities present in the vulnerable version of the CFGS system running on port 3000. Log in to the system first, then run the attacks below.

🍪

Vulnerability (A) — Session Hijacking

Cookies are readable by JavaScript · No session timeout · No session regeneration

Root cause: The server sets httpOnly: false on the session cookie. This means any JavaScript on the page — including scripts injected by an attacker via XSS — can read the full cookie value using document.cookie.

Impact: An attacker who steals the cookie string can paste it into their own browser and impersonate the victim without ever knowing their password. The session also never expires (no maxAge), so a stolen token works indefinitely.

  4. 4

  5. 5

Vulnerability (B) — Privilege Escalation

Backend has no role checks · Role accepted from request body · Frontend-only restrictions

Root cause: Role-based restrictions exist only in the frontend UI (e.g. "Close" buttons are hidden for employees). The backend API endpoints perform no role validation on the server side. Additionally, one endpoint reads the user's role directly from the request body — meaning the client can lie about their role.

Impact: Any authenticated user, regardless of role, can call privileged endpoints directly and perform admin or manager actions.

Attack 1 — Close a ticket without manager role

The PUT /api/tickets/:id/close endpoint has no role check. Any logged-in user can call it. The frontend just hides the button for employees.

fetch('/api/tickets/1/close', { method: 'PUT' }) .then(r => r.json()).then(console.log)

Attack 2 — Delete a ticket by faking admin role in request body

The DELETE /api/tickets/:id endpoint reads role from req.body instead of req.session. Sending { "role": "admin" } in the body bypasses the check entirely.

fetch('/api/tickets/2', { method: 'DELETE', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ role: 'admin' }) // ← lying about role! }).then(r => r.json()).then(console.log)

Attack 3 — Access the admin user list without admin role

GET /api/admin/users only checks if the user is logged in — it does not verify whether they are an admin.

fetch('/api/admin/users') .then(r => r.json()).then(console.log)

Vulnerability (C) — SQL Injection

User input concatenated directly into query string · UNION attack leaks all passwords

Root cause: The search endpoint builds its query by pasting the user's input directly into a string: WHERE title LIKE '%INPUT%'.

An attacker closes the string early with ' then appends a UNION SELECT statement pointing at the users table. The database executes both statements and returns the combined results — including every username and plaintext password.

Impact: Full credential dump. Anyone who can search can extract every account in the system.

Step 1 — Normal search (see how it works first)

The search bar on the dashboard calls GET /api/search?q=TERM. The raw SQL query is shown below the search bar on the dashboard. Try searching for wifi.

GET /api/search?q=wifi → Executes: SELECT ... FROM tickets WHERE title LIKE '%wifi%' OR description LIKE '%wifi%' → Returns: matching tickets only ✓

Step 2 — Injection attack: dump all user passwords

Paste the payload below into the search box on the dashboard, or click Run Attack. The UNION appends a full select from the users table. Usernames appear as ticket titles, passwords appear as descriptions.

' UNION SELECT id,username,password,role,name,name,name FROM users --

The server executes:
SELECT ... FROM tickets WHERE title LIKE '%' UNION SELECT id,username,password,role,name,name,name FROM users --%'

Vulnerability (D) — Ping of Death

No payload size limit on /api/ping · Oversized packet forces full memory allocation · Server crash

Original Ping of Death (network layer): An attacker sends an ICMP "ping" packet whose total size, when all IP fragments are reassembled, exceeds 65,535 bytes — the maximum allowed by the IPv4 specification. The target OS allocates a buffer for the full packet, the buffer overflows, and the system crashes or reboots.

Web application equivalent: This server exposes a POST /api/ping endpoint with no payload size limit. Node.js must buffer the entire request body in heap memory before the handler can run. Sending a large enough payload exhausts available RAM and crashes the process with "JavaScript heap out of memory" — the same outcome as the original attack, just at the application layer instead of the OS.

Impact: Complete server crash, all active users lose their sessions, service is unavailable until the process is manually restarted.
Original (ICMP)
Attacker → fragmented ICMP packets → target OS reassembles → total size > 65,535 bytes → buffer overflow → OS crash
Web Equivalent (HTTP)
Attacker → large POST body → Node.js buffers entire body → heap exhausted → process crash

Step-by-step: How the attack works

  1. 1
    The server registers POST /api/ping using express.raw({ limit: Infinity }) — no size cap at all.
  2. 2
    Attacker sends a massive HTTP POST body to /api/ping (same concept as a giant ICMP packet).
  3. 3
    Node.js must fully buffer the body in heap memory before the route handler can read even a single byte.
  4. 4
    With a large enough payload (try 50 MB+) the Node.js heap is exhausted and the process crashes — all sessions lost, server offline.

Live attacks — run them in sequence

Attack 1: Small ping (64 bytes) — works normally.

POST /api/ping body: 64 bytes of 'A' → server echoes: Pong! (64 bytes read)

Attack 2: Oversized ping (5 MB) — server buffers the whole thing. On the secure system this returns HTTP 413 immediately.

POST /api/ping body: 5,000,000 bytes → vulnerable server reads ALL of it into RAM

Attack 3: Fatal ping (50 MB) — may crash the server process. ⚠ This may take a few seconds and could kill the server.

POST /api/ping body: 50,000,000 bytes → heap exhaustion → process crash
